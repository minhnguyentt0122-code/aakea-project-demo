### F-01.6 Customer Account Profile Management

> *This feature was implied by the registration and order flows but not explicitly specified in v2.0. Added in v2.1.*

**Summary**

Authenticated customers can view and update their account profile (name, email, phone), change their password, manage saved delivery addresses, view their trade account status, and close their account.

**Background / Context**

For premium design buyers — especially trade customers managing multiple project delivery addresses — robust account management is a baseline expectation. Without these APIs, the frontend has no endpoints to render or update the `/account/profile` and `/account/addresses` pages referenced throughout the spec.

**User Story**

As a registered customer, I want to update my personal details, manage my saved delivery addresses, and change my password from my account page so that my account reflects current information.

**Flow — Happy Path**

1. Customer navigates to `/account/profile`. Page loads via `GET /api/v1/account/me`.
2. Customer updates first name, last name, or phone and clicks Save.
3. `PATCH /api/v1/account/me` updates only the supplied fields.
4. Customer navigates to `/account/addresses`. `GET /api/v1/account/addresses` returns saved addresses.
5. Customer adds a new address (POST), edits an existing one (PATCH), deletes one (DELETE), or sets a default (PATCH with `isDefault: true`).
6. Customer navigates to `/account/security` and changes password via `PATCH /api/v1/account/password`. All existing refresh tokens are revoked.

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Customer attempts to change email | Email change requires re-verification: new email receives verification link. Old email remains active until new one is verified. |
| Customer deletes their default address | If other addresses exist, first remaining address promoted to default. If no addresses remain, `defaultAddressId` set to null. |
| Customer requests account closure | Initiates soft-delete flow per F-08.2 (Data Erasure). Account deactivated immediately; erasure scheduled per TC-31. |
| Customer sets a non-existent address as default | 404 returned. |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | PATCH /account/me accepts any subset of updatable fields. Fields not supplied are unchanged. |
| BR-02 | Email change requires re-verification of the new email before it replaces the old one. |
| BR-03 | Password change via PATCH /account/password requires the customer to supply their current password for confirmation. |
| BR-04 | Successful password change revokes all refresh tokens (same as F-01.3 reset). |
| BR-05 | Maximum 10 saved delivery addresses per customer. Attempting to add an 11th returns 422. |
| BR-06 | Exactly one address may be flagged as `isDefault=true` per customer. Setting a new default clears the previous flag atomically. |

**Input Validation Rules**

| Field | Type | Required | Validation Rules | Error Message Shown |
|---|---|---|---|---|
| firstName | String | No (PATCH) | Min 1, max 50 chars. | 'First name must be 50 characters or fewer' |
| lastName | String | No (PATCH) | Same as firstName. | 'Last name must be 50 characters or fewer' |
| phone | String | No | E.164 format (e.g. +12025551234). Max 20 chars. | 'Please enter a valid phone number' |
| currentPassword | String | Yes (for PATCH /password) | Validated against stored bcrypt hash. | 'Current password is incorrect' |
| newPassword | String | Yes (for PATCH /password) | Same rules as registration. Must differ from currentPassword. | 'New password must meet complexity requirements' |
| address fields | Various | Yes (for POST /addresses) | line1 (max 100), line2 (optional, max 100), city (max 100), state/county (max 100), postalCode, country (ISO 3166-1 alpha-2). | Per-field: '[Field] is required' |

**UI / Display Rules**

| Condition / State | UI Component | Display Text | Colour / Style | Available Action |
|---|---|---|---|---|
| Profile saved successfully | Success toast | 'Your profile has been updated.' | Green toast | — |
| Email change pending verification | Info banner | 'A verification link was sent to [newEmail]. Your email will update once verified.' | Blue info banner | 'Resend link' / 'Cancel change' |
| Password changed successfully | Success toast + redirect | 'Password changed. Please log in again.' | Green toast | Redirected to /login |
| Address limit reached (10) | Add address button disabled | 'You have reached the maximum of 10 saved addresses.' | Amber tooltip | Delete an existing address to add a new one |
| Default address badge | Address card | 'Default' | Green badge | — |
| Account closure confirmation modal | Modal | 'Are you sure? This will permanently delete your account and all data.' | Red destructive action modal | 'Cancel' / 'Delete My Account' (requires password) |

**API Contract**

| Method | Endpoint | Description | Request Body / Params | Response |
|---|---|---|---|---|
| **GET** | /api/v1/account/me | Get profile | — | `200: { id, firstName, lastName, email, phone, tradeAccountStatus, createdAt }` |
| **PATCH** | /api/v1/account/me | Update profile | `{ firstName?, lastName?, phone? }` | `200: { user }` |
| **PATCH** | /api/v1/account/email | Request email change | `{ newEmail, password }` | `200: { message }` / `409: email taken` |
| **PATCH** | /api/v1/account/password | Change password | `{ currentPassword, newPassword }` | `200: { message }` / `401: wrong current password` |
| **GET** | /api/v1/account/addresses | List addresses | — | `200: { addresses[] }` |
| **POST** | /api/v1/account/addresses | Add address | `{ line1, line2?, city, state, postalCode, country, isDefault? }` | `201: { address }` / `422: limit reached` |
| **PATCH** | /api/v1/account/addresses/:addressId | Edit address | `{ line1?, city?, postalCode?, isDefault?, … }` | `200: { address }` |
| **DELETE** | /api/v1/account/addresses/:addressId | Delete address | — | `200: { message }` |
| **POST** | /api/v1/account/close | Close account | `{ password }` | `200: { message } — schedules erasure` |
