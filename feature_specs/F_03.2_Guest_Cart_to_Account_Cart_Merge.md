### F-03.2 Guest Cart → Account Cart Merge

**Summary**

When a guest logs in or registers, their session cart — including full configuration detail — is merged into their authenticated account cart atomically.

**Background / Context**

Cart abandonment due to login friction is a major conversion killer, particularly for high-consideration purchases. A premium design customer who has spent time selecting material, finish, and fabric grade options should not lose their selections when they authenticate to complete the purchase.

**User Story**

As a guest who configured and added items before logging in, I want my cart — including all my selected options — to be preserved after I authenticate so I don't have to reconfigure everything.

**Flow — Happy Path**

1. Guest adds configured items (Redis session, TC-12: 30-day TTL).
2. Guest proceeds to checkout → prompted to log in or register.
3. On successful authentication, frontend sends login request with `guest_session_token`.
4. Server fetches guest cart from Redis. Merges into user's DB cart: configurations not in cart added; matching configurations summed (capped at 10/configuration).
5. Redis guest cart key deleted.
6. User sees merged cart. Toast: 'Your cart has been saved to your account.'

**Flow — Edge Cases**

| Edge Case | Expected System Behaviour |
|---|---|
| Guest cart expired (TC-12) before login | Merge silently skipped. Only authenticated cart shown. No error. |
| Quantity sum post-merge exceeds cap | Capped at 10. Toast: 'Some quantities were adjusted.' |
| Guest cart contains configuration now out of stock | Merged into cart but flagged as out of stock on /cart load. |

**Business Rules**

| BR # | Rule |
|---|---|
| BR-01 | Cart merge is atomic (DB transaction) — all items transfer or none. |
| BR-02 | TC-12: Expired guest cart silently skipped — merge failure is non-critical. |
| BR-03 | `guest_session_token` cookie deleted server-side after successful merge. |
| BR-04 | If user has no existing DB cart, merge simply creates cart entries from guest cart. |

**API Contract**

| Method | Endpoint | Description | Request Body / Params | Response |
|---|---|---|---|---|
| **POST** | /api/v1/auth/login | Login with optional guest cart merge | `{ email, password, guestSessionToken? }` | `200: { accessToken, cart (merged) }` |

**TC References:** TC-12 (guest cart 30-day Redis TTL — merge silently skipped if expired)

**Acceptance Criteria**

| # | Criteria | Test Type | Status |
|---|---|---|---|
| AC-01 | Guest items with full configuration detail appear in authenticated cart immediately after login. | E2E | — |
| AC-02 | Same configuration in both carts: quantities summed and capped at 10. | Integration | — |
| AC-03 | Guest Redis cart key deleted after successful merge. | Integration | — |
| AC-04 | Expired guest cart: no error shown, only authenticated cart displayed. | Integration | — |
